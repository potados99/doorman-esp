# Per-Device Config & UI Redesign Brainstorm

**Date**: 2026-04-07
**Status**: Final

## What We're Building

두 가지 변경:

1. **기기별 설정 분리**: 현재 전역 단일 AppConfig를 기기(MAC) 단위로 분리. 감지 파라미터 4개(rssi_threshold, presence_timeout_ms, enter_window_ms, enter_min_count)를 기기마다 다르게 적용. auto_unlock은 전역 유지.
2. **UI를 기기 카드 중심으로 재설계**: 탭 분리(대시보드/설정) 대신, 기기 카드 리스트가 메인 화면. 각 카드에 실시간 상태 + 개별 설정. 로그 뷰어는 디버깅용 보조 도구로 전환.

## Why This Approach

### 기기별 설정이 필요한 이유
- 사람마다 폰 BLE 송신 파워, 위치(주머니/가방), 기기 종류(BLE/Classic)가 달라 최적 RSSI/타임아웃이 다름
- 현재는 한 사람에게 맞추면 다른 사람이 안 잡히거나, 반대로 너무 민감해지는 문제

### 카드 UI가 맞는 이유
- 사용자가 실제로 알고 싶은 건 "누가 무슨 상태인가"이지 로그 텍스트가 아님
- 로그는 "왜 안 되지?" 디버깅할 때만 필요
- 기기 카드에 상태와 설정이 같이 있으면 "로그 보면서 설정 조작" 워크플로우가 불필요해짐

## Key Decisions

### 1. 설정 단위: 기기(MAC) + 별명
- 사람(프로필) 개념 없이 MAC 단위로 설정 저장
- UI에서 구별 위해 기기별 별명(닉네임) 지원
- 기기별 오버라이드가 없으면 하드코드된 기본값 적용 (전역 기본값 UI 없음)

### 2. 기기별 개인화 항목
- rssi_threshold, presence_timeout_ms, enter_window_ms, enter_min_count (감지 파라미터 4개)
- auto_unlock_enabled는 전역 유지
- alias(별명)은 기기별 추가 필드

### 3. UI 구조: 기기 관리 앱

멘탈 모델: "로그 보는 대시보드"가 아니라 **"기기를 추가하고, 상태를 보고, 관리하는 앱"**.

**메인 화면** — 컴팩트 카드 리스트 + 빠른 동작:
```
┌──────────────────────────┐
│ Doorman    [+ 기기] [📋] │
├──────────────────────────┤
│ ┌▌Profession     -49dBm┐ │  ← 녹색 보더 = 재실
│ └──────────────────────┘ │
│ ┌▌엄마 아이폰       2/3 ┐ │  ← detecting 보더
│ └──────────────────────┘ │
│ ┌│Galaxy S24           ┐ │  ← 회색 보더 = 퇴실
│ └──────────────────────┘ │
│                          │
│ ── 빠른 동작 ──          │
│ [🔘 문 열기]             │
│ 자동열림          [OFF]  │
└──────────────────────────┘
```
- 카드 1줄 높이: 보더 색상으로 상태, 별명 + 보조 정보(RSSI or 카운트)만 표시
- 기기 5~6개가 스크롤 없이 한눈에 보임

**카드 탭 → 기기 상세 모달**:
```
┌──────────────────────────┐
│ Profession           [✕] │
│ ─────────────────────── │
│ 상태: 재실               │
│ RSSI: -49dBm             │
│ 감지: 3/3 (5000ms)       │
│                          │
│ ── 설정 ──               │
│ 별명: [Profession    ]   │
│ RSSI 임계값: [-70  ]     │
│ 퇴실 타임아웃: [15000]   │
│ 진입 윈도우: [5000 ]     │
│ 진입 최소 관측: [3  ]    │
│ [저장]          [삭제]   │
└──────────────────────────┘
```

**[+ 기기] → 페어링 모달** (닫기 = 페어링 종료):
```
┌──────────────────────────┐
│ 기기 추가            [✕] │
│ ─────────────────────── │
│                          │
│       (○ 스피너)         │
│     페어링 대기 중...     │
│                          │
│ 기기의 블루투스 설정에서  │
│ 아래 이름을 찾아 연결하세요│
│                          │
│  BLE: "Doorman"          │
│  Classic: "Doorman SPP"  │
│  PIN 요청 시 1234 입력   │
│                          │
│ ─────────────────────── │
│ 이 창을 닫으면 페어링이  │
│ 종료됩니다.              │
└──────────────────────────┘
```
- ✕ 또는 밖 터치 → 페어링 종료 API 호출 → 모달 닫힘 → 기기 목록 갱신
- 페어링 중 `bt: MAC paired 기기명` 로그 수신 시 모달 내에 "기기명 연결됨!" 표시 (여러 기기가 연속 페어링될 수 있으므로 리스트로 누적 표시)

**기기 0개 — 온보딩 안내:**
```
┌──────────────────────────┐
│ Doorman    [+ 기기] [📋] │
├──────────────────────────┤
│                          │
│   등록된 기기가 없습니다  │
│   [+ 기기]로 시작하세요   │
│                          │
└──────────────────────────┘
```

**전체 구조:**
- 탭: [메인] [관리]. 메인=기기 카드+빠른동작. 관리=Danger Zone(OTA, WiFi, 계정, 재부팅)
- 헤더: [+ 기기] = 페어링 시작, [📋] = 로그 전체화면 오버레이 (기존 필터/태그/텍스트검색 유지)
- 데스크탑: 모바일과 동일한 단일 컬럼(max-width: 520px). 반응형 분기 없음

### 4. 로그 포맷: bt/sm 태그를 구조적으로 고정

별도 채널/프로토콜/태그 없이, bt와 sm 태그 로그 자체가 구조적 데이터 소스.

**bt 포맷** — MAC 고정 위치, BLE/Classic 구분 안 함:
```
bt: 84:2F:57:A0:C4:EA rssi=-49
bt: 84:2F:57:A0:C4:EA rssi=0           ← Classic (RSSI 미지원, 카드에서 0이면 비표시)
bt: 84:2F:57:A0:C4:EA paired Profession ← 페어링 완료 (기기 이름 포함, 띄어쓰기 가능)
```

**sm 포맷** — 4개 키워드, 각각 역할이 다름:
```
sm: 84:2F:57:A0:C4:EA detecting 2/3 5000   ← 감지 진행 (카운트/분모 윈도우ms)
sm: 84:2F:57:A0:C4:EA present              ← 재실 확정 (숫자 없음, detecting이 이미 보여줌)
sm: 84:2F:57:A0:C4:EA absent 15000         ← 퇴실 (타임아웃ms, 과정 없이 한 방이라 여기서 표시)
sm: 84:2F:57:A0:C4:EA unlock               ← 문열림 판정 (SM 상태 전환, 억제 여부는 SM Task 영역)
```

**접근 시나리오 전체 흐름:**
```
bt: 84:2F:57:A0:C4:EA rssi=-85             ← threshold 미달, SM 무시
bt: 84:2F:57:A0:C4:EA rssi=-65             ← threshold 통과
sm: 84:2F:57:A0:C4:EA detecting 1/3 5000
bt: 84:2F:57:A0:C4:EA rssi=-60
sm: 84:2F:57:A0:C4:EA detecting 2/3 5000
sm: 84:2F:57:A0:C4:EA detecting 1/3 5000   ← 윈도우에서 오래된 관측 빠짐 (카운트 감소 가능)
bt: 84:2F:57:A0:C4:EA rssi=-55
sm: 84:2F:57:A0:C4:EA detecting 2/3 5000
sm: 84:2F:57:A0:C4:EA detecting 3/3 5000
sm: 84:2F:57:A0:C4:EA present              ← 재실 확정
sm: 84:2F:57:A0:C4:EA unlock               ← SM 문열림 판정 (항상 찍힘, 실제 억제 여부는 SM Task 영역)
...시간 경과...
sm: 84:2F:57:A0:C4:EA absent 15000         ← 퇴실
```

**설계 원칙:**
- 이 포맷 이외의 bt/sm 로그(SPP, advertising 등)는 UI 업데이트에 사용하지 않음 (정규식에 매칭 안 되므로 자동 무시)
- detecting은 카운트 변할 때만 찍음 (매 tick 아님). 감지 진행 상태 표시용
- present/absent/unlock은 SM 상태 전환 이벤트
- paired는 페어링 완료 시 1회. 페어링 모달에서 파싱하여 "기기명 연결됨!" 피드백 표시. 기기명을 별명 기본값으로 자동 저장 (NVS)
- "안 보이기 시작함" 표시(진녹색)는 브라우저 클라이언트 타이머로 처리 (bt: 라인이 일정 시간 안 오면 전환)
- Classic rssi=0은 카드에서 비표시 처리

**브라우저 파싱:**
```js
var bt = line.match(/bt: ([0-9A-F:]{17}) rssi=(-?\d+)/);
var pair = line.match(/bt: ([0-9A-F:]{17}) paired (.+)/);
var sm = line.match(/sm: ([0-9A-F:]{17}) (detecting|present|absent|unlock)/);
```

**카드 상태 표현 — 리플 애니메이션 + 색상:**
- `detecting` → 보더에서 바깥으로 리플 이펙트 (심장박동처럼 뛱...뛱...). 카운트 올라갈수록 리플 간격 짧아짐 (1/3 느리게, 3/3 빠르게). 카운트 텍스트도 표시.
- `present` → 강한 리플 한 방 퐁! + 녹색 보더 유지
- present인데 bt: 라인 ~3초 부재 → 진녹색 (클라이언트 타이머, 리플 없음)
- `absent` → 회색, 리플 없음
- `unlock` → 문열림 인디케이터 (잠시 표시)
- CSS `@keyframes` ripple + `animation-duration`을 카운트에 따라 조절하여 구현

### 5. 초기 상태 로드: REST 합성 응답
```
GET /api/devices 응답 흐름:

1. bt_get_bonded_devices()     → 페어링된 MAC 목록
2. config_get_device(mac)      → 별명, 개별 감지 파라미터
3. sm_get_snapshots()          → 현재 상태 (재실/퇴실, RSSI, last_seen 등)
4. 합쳐서 응답
```

### 6. SM 상태 외부 노출: config_service 역방향 패턴
- StateMachine에 `dump_states()` 메서드 추가
- SM Task가 매 tick 후 mutex 보호 전역 스냅샷 갱신
- HTTP 핸들러가 mutex 잡고 memcpy로 읽음
- `app_config_get()` 패턴과 동일, 방향만 반대 (SM 쓰기 → HTTP 읽기)
- 매 tick(100ms) memcpy ~900바이트 — ESP32에서 무시할 수준

### 7. 기기별 설정 저장: NVS
- MAC 12hex를 키로, 설정 구조체 전체를 blob으로 직렬화
- 예: namespace `"dev"`, key `"842F57A0C4EA"`, value = blob

## Technical Challenges

구현 시 핵심 기술 과제 3가지:

1. **Config에 기기별 설정 배열 + 빠른 조회 + NVS 영속화**: 현재 flat 전역 config → MAC 키 기반 배열로 확장. tick마다 MAC으로 config lookup (O(n), 기기 15대 이하면 충분). NVS는 MAC 12hex 키 + blob 직렬화.
2. **SM 상태 외부 조회**: StateMachine에 dump 메서드 추가, SM Task가 매 tick 후 mutex 보호 스냅샷 갱신, HTTP 핸들러가 읽기. config_service 역방향 패턴.
3. **브라우저 로그 스트림 파싱 → 실시간 카드 갱신**: bt/sm 로그 포맷을 고정하고, 클라이언트에서 정규식으로 MAC+상태 추출하여 DOM 업데이트. 초기 상태는 REST, 이후는 WebSocket 로그 파싱. 이 포맷 이외의 로그는 자동 무시.
